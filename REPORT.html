1 — Refresher

이 프로젝트가 뭐였지: 30초 리마인드

Polymarket의 BTC 15분 Up/Down 마켓(15분마다 새 slug가 열리는 바이너리 마켓)을 자동매매하는 봇. 1초 tick으로 호가를 읽고, 전략이 진입/청산 신호를 내면 executor가 주문을 낸다. 2026-07-06 리팩토링으로 공유 코어 엔진 + 전략 플러그인 구조가 됐고, 레거시 3개 폴더는 읽기 전용 아카이브다.

slug_loop
1초 tick · 15분 slug 롤링 · 호가 수집 (source)
strategy
on_event → intent 결정. 포지션은 account에서 읽기만 (decision)
executor
sim/live 주문 집행. 무상태 (execute)
account
포지션의 단일 진실(SOT). 체결 결과는 on_trade로 전략에 피드백
logger / printer
events·trades·snapshots CSV, append + run_id (sink)
polybots/
├── core/          공유 엔진 — adapter(재시도) · slug_loop · executor(sim/live) · account(sim/live)
│                  logger(append+run_id) · printer · runner(CLI)
├── strategies/    base(인터페이스) · threshold · ma_breakout     ← 1 전략 = 1 모듈
├── configs/       threshold.json · ma_breakout.json
├── backtest/      data_prep · run_grid(병렬) · engine(정밀 리플레이) · sweep_threshold · README
├── logs/          런타임 CSV (append, run_id로 실행 구분)
├── CLAUDE.md · SPEC.md · ANALYSIS.md · STATUS.md · REPORT.html(본 문서)
└── polybots_MA/ · polybots_pre/ · polybots_backtest/   (읽기 전용 아카이브 — 새 작업 금지)

실행:  python -m core.runner --strategy threshold --mode sim     ← sim이 기본값

안전 규칙 절대 준수

  • --mode live는 명시적으로 결정했을 때만 — 실제 지갑으로 실주문이 나간다. config로는 live가 켜지지 않고 CLI 플래그만이 결정.
  • .env에 지갑 개인키 — 읽기/출력/커밋/전송 금지 (.gitignore 등록됨).
  • 전략/executor 수정 후에는 반드시 sim으로 먼저 검증.

설계 불변식 수정 시 유지

  • 단방향 스트림: source → decision → execute → SOT → sink
  • Account = 포지션 SOT. 전략의 낙관적 포지션 설정 금지.
  • 전략 내부 카운터/락은 on_trade(filled) 확인 후에만 갱신.
  • 시간 판단은 time_left_sec 기준 (wall-clock ts는 로그용).
2 — Strategies

전략 현황판: 규칙 · 현행 파라미터 · 판정

threshold 주력 후보 — sim 검증 중

구 pre 전략. "이미 우세한 쪽이 그대로 끝난다"에 베팅하는 모멘텀 추종.

  • 진입: 남은시간 450→80초 창에서 우세측 ask ≥ 0.8 매수 (단, 가격이 entry_cap 0.9 초과면 포기)
  • 손절: 진입가 대비 −0.06 하락 시 market 청산
  • 익절: 0.99 limit (← 스윕 결과로 0.98에서 상향한 유일한 변경)
  • 재진입: slug당 최대 2회, 직전 120초 낙폭이 −0.15보다 나쁘면 재진입 차단(dd 필터)
  • 강제 청산: 만기 50초 전

비용 반영 백테스트: 전 기간 +$27.2 · 검증셋 +$2.7 · MDD −12.7 · 72조합 스윕에서 현행 파라미터가 2위(1위와 tp만 차이) — 강건한 최적점.

ma_breakout 라이브 부적합 판정

mid 가격의 MA(300틱) 상향돌파 + 가격 ≤ cap 0.5일 때 매수, MA 하향돌파 시 청산(체결까지 매 tick 재발행), tp 0.98 limit.

  • 무비용 백테스트 +$63.9 → 비용 반영 시 −$5.2, 검증셋 −$29.9
  • 3,600조합 전수 탐색에서 학습·검증 동시 플러스 단 1개 (경제성 없음)
  • 헤어컷 2센트에 손익이 반전하는 비용 취약 구조 (거래 수가 threshold의 3배)

재검토 조건: 비용 구조 자체를 바꾸는 방향(체결 수 축소, limit 매도)일 때만. 파라미터 튜닝으로는 살아나지 않음이 확인됨.

⚠️ 단서: threshold의 검증셋은 18 slugs(4시간)뿐. +$2.7은 "생존" 신호이지 수익성 확정이 아니다 — 그래서 P0(sim 30 slugs 축적 → 재평가)이 지금의 관문이다.

E — Verdict

"기존 모델은 완벽하게 최적화되어 있었나?"

MA breakout 아니오

기존 최적값은 무비용 가정의 산물. 비용 반영 시 엣지 소멸, 어떤 파라미터 조합도 out-of-sample에서 생존하지 못함. 헤어컷 2센트에 손익 반전하는 비용 취약 구조. 현 형태로는 라이브 재투입 부적합.

threshold 사실상 예

72조합 중 2위 — 이미 준최적이었고 주변 파라미터가 모두 견고. 비용 반영 후에도 학습·검증 모두 플러스. 유일한 개선(tp 0.99)을 config에 반영 완료. sim 검증을 거쳐 주력 후보.

단서: 검증셋은 18 slugs(4시간)뿐이다. threshold의 검증 +$2.7은 "생존" 신호이지 "수익성 확정"이 아니다 — 그래서 다음 단계가 sim 데이터 축적(P0)이다.

F — Roadmap

로드맵 상세 (라이브 연결은 최후순위)

  1. P0 · sim 검증 재개 필요 — threshold rolling sim으로 slug 30개+ 축적(누적으로 충분, 연속 불필요), 진입/청산/slug 전환/로그 무결성 확인. 수집된 events.csv는 자동으로 백테스트 데이터가 된다 (data_prep.py가 통합).
  2. P1 · 모델 업데이트 사이클 환경 구축 완료 — 수집 → data_prep → 그리드/스윕 → engine 정밀 검증 → config 반영. 신규 전략은 strategies/에 추가만 하면 엔진이 바로 백테스트 가능. 신규 데이터 30 slugs+ 확보 시 threshold 재평가가 첫 사이클.
  3. P2 · 실행 품질 — 라이브 손실 요인 제거: 매도 스윕의 메인 루프 블로킹(≤10초) 스레드 분리, exit_tp를 limit 방치가 아닌 스윕 경로로(3/3 세션의 FAK no-match 익절 실패 사례), 주문 전 USDC 실잔고 가드. sell_dust 재시도 정책은 리팩토링에 반영됨 — 라이브 검증만 남음.
  4. P3 · 인프라 — tick당 HTTP 4회 순차 → 병렬화 또는 CLOB WebSocket, slug 경계 404 폴백(로컬 시계 레이스), 서버 상시 가동 + 로그 로테이션.
  5. P2·P3와 병행 가능 · 전략 연구 — threshold 개량(진입 시간창·dd 필터 재탐색), MA는 비용 구조를 바꾸는 방향(체결 수 축소, limit 매도)에서만 재검토, Binance ATR 데이터 결합(SPEC ver5). ETH/SOL/XRP는 config의 slug prefix만 바꾸면 확장 가능.
  6. P4 · (최후) 라이브 재개 — 기준: sim 30 slugs+ 무결 + 현실화 백테스트 기대값 플러스 유지 + P2 실행 품질 완료. 충족 시에도 소액부터.
