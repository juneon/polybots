2. 문제 진단 — 왜 이 작업인가

2.1 전략 이름 불일치

같은 전략이 코드·설정·로그·문서에서 ma_breakout으로, 대화·분석에서는 "MA"로 불린다. run_id(20260706_235841_ma_breakout_sim)에도 박혀 있어 UI 집계 키도 갈라진다. → 코드·설정·문서는 전부 ma로 통일하되, 과거 로그 CSV(수집 자산, events.csv에만 2.4만 행)는 불변 원칙(append-only)에 따라 건드리지 않고 읽는 쪽(ui/metrics.py)에서 ma_breakout → ma로 정규화한다. sim 계좌 파일은 runner 시작 시 자동 이관(기존 루트→state 이관과 같은 패턴).

2.2 완전/미완성 slug 미구분 — 백테스트 왜곡

엔진은 slug 종료 시점의 마지막 bid를 "만기 정산가(≈0 또는 1)"로 간주해 강제 청산한다. 완전한 slug에서만 참인 가정이다. 봇을 중간에 껐다 켠 slug는 마지막 bid가 중간 가격(예: 0.6)이라 실제로는 일어날 수 없는 손익이 계산에 섞인다. 지금까지의 모든 그리드/스윕(어제의 144조합 포함)이 이 미완성 slug 43개를 포함한 채였다.

2.3 robust config 검증 절차의 한계

현재는 train(1~2월)/val(3/3+sim) 분리 + "val 음수 탈락"까지는 있으나, 어제 config 교체(0.85/0.12/0.98/t450)는 mar03 −0.95로 경계선 통과였고, 파라미터 이웃의 안정성(plateau 여부)은 보지 않았다. 목표: 다른(미래) 시계열에 대해 테스트하기 전에, 지금 데이터 안에서 견고성을 최대한 검증한 config를 두 전략 모두에 대해 도출.

complete = ① 시작 커버: 첫 quote의 time_left_sec ≥ 870 (slug 개장 30초 내 관측 시작) AND ② 종료 커버: 마지막 quote의 time_left_sec ≤ 15 (만기 15초 전까지 관측 → 마지막 bid ≈ 정산가) AND ③ 연속성: 내부 최대 갭 ≤ 60초 (중간에 껐다 킨 흔적 없음)

민감도 확인: 기준을 (850/30/60), (880/10/30)으로 흔들어도 완전 slug 수는 200~201로 안정 — 경계에 걸친 slug가 거의 없어 임계값 논쟁의 실익이 없다. 미완성 43개의 사유: 늦은 시작 21 / 이른 종료 20 / 중간 갭 5 (중복 있음).

5. 이번에 확정하는 규칙 (제안)

  1. 이름: 전략 정식명은 threshold, ma. 과거 기록은 불변, 읽기 시점 정규화.
  2. 완전성: complete = 시작 tleft ≥ 870 ∧ 종료 tleft ≤ 15 ∧ 최대 갭 ≤ 60s. 백테스트 기본값은 complete-only.
  3. robust config: train 점수 + val 비음수 + 이웃 plateau. 세 가지를 모두 만족해야 configs/ 반영.
  4. 보고서: reports/<YYYYMMDD>_<주제>_{plan|result}.html, git 추적 (집↔회사 공유). 기존 문서 5개 체계는 불변.

6. 리스크 / 주의

  • 과거 결과와의 단절 — complete-only 전환으로 어제까지의 수치(sim −13.2 등)와 직접 비교가 안 됨. 전환 시점에 신·구 기준 수치를 한 번 나란히 기록해 브리지를 남긴다.
  • 이름 변경 중 실행 충돌 — 확인 결과 이 머신에 실행 중인 봇 없음(logs/ctl 부재). 집 PC도 pull 전 봇 정지 필요 — 사후 보고서에 명시.
  • sim 이월 포지션 버그(P2)는 이번 범위 밖 — 백테스트 엔진은 slug별 리셋이라 무관하지만, sim 실측 로그의 왜곡(threshold −3.5~−8.8 과소)은 여전하므로 sim 소스 해석 시 감안.
  • ma 그리드는 3,600조합 × 리플레이 — 풀 그리드는 10분+ 소요. 완전 slug 필터로 데이터가 줄어 다소 단축 예상.
