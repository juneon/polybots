1. 프로젝트가 무엇인가

Polymarket의 BTC 15분 Up/Down 바이너리 마켓에서 favorite 사이드를 사고 파는 자동매매 봇. 15분마다 새 마켓(slug)이 열리고 만기에 0/1로 정산되는 구조라, 전략은 "언제 진입할 가치가 있는 가격/경로인가"와 "언제 포기(손절)할 것인가"의 게임이다.

구성내용상태
core/공유 엔진 — adapter(Gamma/CLOB) → slug_loop(1s tick) → strategy → executor(sim/live) → account(SOT) → logger/printer. 단방향 스트림, Account가 포지션 단일 진실안정

4.2 ma — 판정 보류 유지 (한 번도 교체 없음)

  • 현행 cap0.5/ma300/tc0/tp0.98. 24k 그리드(07-18)와 sim 60 부분 그리드(07-19) 모두 D23 ③(이웃 plateau) 미달 — ma_len 240/480 절벽, cap·slope 이웃 부호 반전.
  • 결정적 문제: 후보 영역 24콤보 전부 mar03 음수 — val 양수가 100% sim 기여. 유일한 라이브 out-of-sample에서 전패하는 구조가 불변.
  • 성과: slope 필터(H1) 검증 — 매치드 페어 6,048쌍에서 train·val 동시 개선, top100 전부 slope 사용. 후보 A(cap0.5/ma300/slope−0.005)·B(현행+slope−0.005, val +12.58)와 cap0.6/tc2 서브영역(val +17~22)을 차기 관찰 대상으로 등록.
  • MA sim 재가동은 P3 recorder 분리(시세 이중 기록 회피)와 연계해 별도 결정.

4.3 연구에서 얻은 지식 (파라미터보다 오래가는 것)

  1. 시장은 캘리브레이션돼 있다 — 손절 시점 bid ≈ 실제 회복 확률(실현 회복률 68% 일치). threshold는 0.98/enter−0.12 박스에서 ~52/48 확률게임을 왕복비용 ~$0.02 내며 반복하는 구조. exit 미세조정으로 EV가 안 바뀜 → 레버는 ① 진입 알파 ② 왕복비용 ③ 레짐 순.
  2. 엣지는 가격이 아니라 도달 경로에 있다 — 0.85 돌파 후 15초 미만 스파이크 진입은 78%가 무엣지, 15~180초 유지 후 진입은 janfeb·sim 양쪽에서 양수. → enter_stable_sec의 탄생, 07-19 채택 config의 핵심.
  3. 휩쏘 71%는 이상 현상이 아니다 — 브리치 순간의 특징으로 휩쏘/진짜 붕괴를 판별할 수 없음(중앙값 완전 동일 — 중요한 부정적 결과). 대응은 판별이 아니라 노출 시간 축소(t180 계열)와 확인 지연(confirm).
  4. MA의 독소는 상승 MA 추격 진입 — 하락 MA를 뚫는 딥 바운스(slope ≤ −0.005)만 유효. 간이 스크리닝 → 실엔진 24k 그리드에서 재현까지 완료.
  5. 미완성 slug는 판정을 왜곡한다(D22) — threshold −$5.4 왜곡 / ma +$7.6 부풀림을 분리해냄.
  6. D23 3중 기준은 실제로 작동한다 — 07-19 score 전체 1위가 전 축 이웃 절벽으로 기각(나이프엣지), 07-14 후보 영역이 표본 확대만으로 소멸(H3). 소표본 후보는 영역이 아니라 스파이크였다.
  7. 레짐은 뒤집힌다 — janfeb은 favorite 저평가(+EV), 7월 sim은 전 밴드 과대평가(−EV). 파라미터 교체보다 표본 확충이 우선이었던 이유.

6. 인프라·프로세스 자산

  • Control UI (Phase A~D): 봇 시작/정지·수집 게이지(완전 slug 기준), PnL/equity 집계((전략,모드) 분리 — sim/live 절대 비합산), config 폼(검증+백업+diff, account 잠금), 백테스트 실행/결과 비교/1위 반영. Phase E(live 3단계 가드)만 잔여.
  • 백테스트 파이프라인: 실전략 리플레이 단일 엔진 → 그리드/스윕 fan-out. 증분 체크포인트(콤보별 즉시 저장, 같은 --out 재실행으로 이어짐 — 07-18 강제종료에서 실전 검증). CLI도 자동 아카이브.
  • 로그 체계: append + run_id, events는 UTC 일자별 로테이션 + data_prep 파일별 파싱 캐시 (43MB 단일 파일 병목 해소, 4.0s→2.2s).
  • 멀티머신 워크플로(D20): 수집은 한 번에 한 머신, 떠나기 전 정지→커밋→push. 07-16 사고의 교훈: push를 건너뛰면 이틀 뒤 같은 작업을 두 번 하게 된다 — 24k 그리드(~2.5h)와 P2 구현이 실제로 중복됐고, 병합 비용까지 지불. 절차는 문서가 아니라 습관이 지킨다.
  • 판정 프로토콜: 가설 사전 등록(plan 리포트) → 엔진 스윕 → D23 3중 기준 → result 리포트(민감도 차트 포함) + WORKLOG. 간이 스크리닝은 채택 근거로 쓰지 않음.
